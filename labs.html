Break & Fix Labs

10 hands-on scenarios covering all CKA exam domains. Each lab breaks a real cluster failure; you diagnose and fix it using only kubectl and ssh.

Lab 01 of 10
Node NotReady
Mental model Node is like a worker who clocked out — kubelet is the time card. No heartbeat, no work.
What breaks
A worker node stops heartbeating to the control plane. kubelet service stops running on the node.
How it manifests
kubectl get nodes shows NotReady. Pods on that node are evicted to other nodes.
Scenario: Worker node is NotReady. SSH in, find kubelet crashed, restart it via systemd. Verify node returns to Ready.
bash — diagnose
copy
kubectl get nodes
kubectl describe node <node-name>
ssh <node-ip>
sudo systemctl status kubelet
sudo systemctl restart kubelet
Lab 02 of 10
Broken Service
Mental model Service selector is a postal address. Pod labels are the house number. Mismatch = mail never arrives.
What breaks
Service selector labels don't match pod labels. Service can't find its endpoints.
How it manifests
kubectl get endpoints <svc> shows <none>. Traffic timeout when you connect to the service.
Scenario: Service web-svc gets connection refused despite pods running. Fix the service selector or pod labels to match. Verify endpoints appear and traffic works.
bash — diagnose
copy
kubectl get svc web-svc
kubectl get endpoints web-svc
kubectl describe svc web-svc
kubectl get pods --show-labels
kubectl edit svc web-svc
Lab 03 of 10
CrashLoopBackOff
Mental model Container starts, looks for config file, finds nothing, crashes. Kubernetes retries—same wall, same crash.
What breaks
Pod tries to read a missing configuration file. Container exits non-zero immediately on start.
How it manifests
Pod status shows CrashLoopBackOff. Restart count keeps increasing. Previous logs show the crash.
Scenario: Pod app-server crashes because config file /etc/app/config.yaml is missing. Mount the ConfigMap or fix the pod command. Verify pod runs and stays Running.
bash — diagnose
copy
kubectl logs <pod> --previous
kubectl describe pod <pod>
kubectl get cm
kubectl edit pod <pod>
Lab 04 of 10
Scheduler Down
Mental model Pods are planes waiting for a gate assignment. No scheduler = planes circle forever, never land.
What breaks
kube-scheduler static pod manifest missing from control plane. Scheduler pod can't start.
How it manifests
New pods stuck in Pending. No scheduling events. Existing pods keep running.
Scenario: Scheduler manifest deleted from /etc/kubernetes/manifests/. SSH to control plane, restore the manifest. Kubelet recreates the pod. Verify new pods now schedule.
bash — diagnose
copy
kubectl get pods -n kube-system | grep scheduler
kubectl describe pod <scheduler> -n kube-system
ssh <control-plane-ip>
ls /etc/kubernetes/manifests/
Lab 05 of 10
Certificate Expiry
Mental model Cluster components check each other's certificates like a border agent checks passports. Expired = denied.
What breaks
API server TLS certificate is expired or about to expire. Components can't verify each other.
How it manifests
kubectl auth errors, nodes NotReady, kubelet can't reach apiserver. Cluster communication fails.
Scenario: Inspect TLS certificate expiry using openssl and kubeadm. Document the expiry date and steps to renew. This is a detection + documentation task, not a fix.
bash — diagnose
copy
ssh <control-plane>
kubeadm certs check-expiration
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -noout -dates
Lab 06 of 10
NetworkPolicy
Mental model By default every pod talks to every pod. NetworkPolicy is the bouncer—you only get in if you're on the list.
What breaks
NetworkPolicy denies traffic that should be allowed. Selector or port mismatch in the policy rule.
How it manifests
Pod-to-pod communication times out or is refused. Logs show connection timeout (not refused—just hangs).
Scenario: Pod client-app can't reach server-app. NetworkPolicy is in place but has a misconfigured selector or missing port rule. Fix the policy and verify traffic flows.
bash — diagnose
copy
kubectl get networkpolicies
kubectl describe networkpolicy <name>
kubectl exec <pod> -- curl <target-pod>:8080
kubectl edit networkpolicy <name>
Lab 07 of 10
PVC Pending
Mental model PVC is your rental contract. PV is the apartment. If no apartment matches your requirements, you wait indefinitely.
What breaks
PVC selector or StorageClass doesn't match any available PV. PVC stays Pending forever.
How it manifests
PVC status shows Pending. Pod using the PVC is also Pending. No PV gets bound.
Scenario: PVC data-pvc stuck in Pending. StorageClass name in PVC doesn't match any StorageClass. Fix the PVC spec or create the matching StorageClass. Verify PVC binds to a PV.
bash — diagnose
copy
kubectl get pvc
kubectl describe pvc <pvc-name>
kubectl get storageclass
kubectl get pv
kubectl edit pvc <pvc-name>
Lab 08 of 10
RBAC Denied
Mental model ServiceAccount is your employee badge. Role is the access level. Without the right binding, the door scanner says no.
What breaks
RoleBinding is missing or refers to wrong ServiceAccount/Role. Pod can't call API.
How it manifests
Pod logs show "forbidden" error when calling apiserver. kubectl auth can-i shows false.
Scenario: Pod app tries to list pods in the namespace via API. Gets forbidden. ServiceAccount exists but RoleBinding is missing. Create the RoleBinding to grant the role to the ServiceAccount.
bash — diagnose
copy
kubectl get serviceaccount
kubectl get role
kubectl get rolebinding
kubectl auth can-i list pods --as=system:serviceaccount:default:app
kubectl create rolebinding <name> --role=<role> --serviceaccount=default:<sa>
Lab 09 of 10
etcd Restore
Mental model etcd is the cluster's memory. Restore is like loading a brain backup—everything snaps back to that point in time.
What breaks
Data was deleted from etcd (e.g., namespace deleted). etcd currently has the wrong state.
How it manifests
Namespace or resources gone. etcd is running fine. You have an etcd snapshot from before the deletion.
Scenario: A namespace was accidentally deleted. Restore etcd from a snapshot backup to recover the deleted resources. Verify the namespace and its pods are back.
bash — diagnose
copy
ssh <control-plane>
sudo systemctl stop kubelet
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
sudo etcdctl snapshot restore /path/to/snapshot.db \
  --data-dir=/var/lib/etcd-restore
sudo rm -rf /var/lib/etcd && mv /var/lib/etcd-restore /var/lib/etcd
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/
sudo systemctl start kubelet
Lab 10 of 10
Kubelet Misconfigured
Mental model Kubelet is the engine of each node. Wrong config = engine misfires. Pods won't start, node reports garbage.
What breaks
Kubelet config file path is wrong or the config itself has invalid settings. Kubelet can't start or runs with broken behavior.
How it manifests
Node is NotReady or in Unknown state. Kubelet is crashed or running incorrectly. Pods can't be created on that node.
Scenario: Node kubelet is misconfigured. The --config flag points to a non-existent file or has syntax errors. Find the correct config path or fix the kubeconfig. Restart kubelet and verify node returns to Ready.
bash — diagnose
copy
ssh <node-ip>
sudo systemctl status kubelet
sudo journalctl -u kubelet -n 50
ps aux | grep kubelet
sudo cat /etc/kubernetes/kubelet.conf
sudo systemctl restart kubelet
