Break & Fix Scenarios

Break & Fix Labs

40 hands-on scenarios covering all CKA exam domains. Each lab breaks a real cluster failure; you diagnose and fix it using only kubectl and ssh.

Lab 01 of 40
Node NotReady
Mental model Node is like a worker who clocked out — kubelet is the time card. No heartbeat, no work.
What breaks
A worker node stops heartbeating to the control plane. kubelet service stops running on the node.
How it manifests
kubectl get nodes shows NotReady. Pods on that node are evicted to other nodes.
Scenario: Worker node is NotReady. SSH in, find kubelet crashed, restart it via systemd. Verify node returns to Ready.
bash — diagnose
copy
kubectl get nodes
kubectl describe node <node-name>
ssh <node-ip>
sudo systemctl status kubelet
sudo systemctl restart kubelet

What breaks

How it manifests

Scenario: Service web-svc gets connection refused despite pods running. Fix the service selector or pod labels to match. Verify endpoints appear and traffic works.

Lab 09 of 40
etcd Restore
Mental model etcd is the cluster's memory. Restore is like loading a brain backup—everything snaps back to that point in time.
What breaks
Data was deleted from etcd (e.g., namespace deleted). etcd currently has the wrong state.
How it manifests
Namespace or resources gone. etcd is running fine. You have an etcd snapshot from before the deletion.
Scenario: A namespace was accidentally deleted. Restore etcd from a snapshot backup to recover the deleted resources. Verify the namespace and its pods are back.
bash — diagnose
copy
ssh <control-plane>
sudo systemctl stop kubelet
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
sudo etcdctl snapshot restore /path/to/snapshot.db \
  --data-dir=/var/lib/etcd-restore
sudo rm -rf /var/lib/etcd && mv /var/lib/etcd-restore /var/lib/etcd
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/
sudo systemctl start kubelet

Audit Logging
Mental model Audit log is the security camera. No camera = no recording of who did what. Enable it on the API server, define the policy, verify the tape rolls.
What breaks
API server audit logging not configured. --audit-policy-file and --audit-log-path flags missing from apiserver manifest.
How it manifests

Scenario: Configure API server audit logging. Create an audit policy file and add the --audit-policy-file and --audit-log-path flags to the kube-apiserver static pod manifest.
bash — fix
copy
ssh <control-plane>
# create audit policy

apiVersion: audit.k8s.io/v1
kind: Policy
rules:
- level: Metadata
EOF
# add to kube-apiserver manifest:
# --audit-policy-file=/etc/kubernetes/audit-policy.yaml
# --audit-log-path=/var/log/kubernetes/audit.log
sudo vi /etc/kubernetes/manifests/kube-apiserver.yaml

Mental model etcd snapshot is a database export. No backup = no recovery. Run it before every change. Store it somewhere safe.

bash — backup
copy

Namespace Isolation
Mental model Default Kubernetes = open office, everyone hears everything. NetworkPolicy = glass walls. Add them to isolate tenants from each other.
What breaks

How it manifests
Pods in namespace A can curl pods in namespace B without restriction. Security boundary doesn't exist.
Scenario: Create a NetworkPolicy that enforces namespace isolation—deny all ingress from other namespaces while allowing intra-namespace traffic. Verify cross-namespace traffic is blocked.
yaml — policy
copy
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-cross-namespace
  namespace: <target-namespace>
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector: {}  # only same namespace

Resource Limits
Mental model Requests = the seat reservation. Limits = the hard cap. No limits = one pod can eat the whole buffet and starve every other guest.

How it manifests

Scenario: Pods have no resource limits. Create a LimitRange in the namespace and add requests/limits to the deployment. Verify pods are constrained and namespace defaults apply to new pods.
bash — fix
copy
kubectl top pods

kubectl create -f - <<EOF
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
spec:
  limits:
  - type: Container
    default: {cpu: 200m, memory: 128Mi}
    defaultRequest: {cpu: 100m, memory: 64Mi}
EOF

Priority & Preemption
Mental model PriorityClass is first class vs coach. High-priority pod gets a seat even if the flight is full—lower-priority pod gets bumped.

How it manifests

Scenario: Critical pods are stuck behind batch workloads. Create a PriorityClass with high value and assign it to the critical deployment. Verify critical pods preempt and schedule first.
bash — fix
copy
kubectl get priorityclass

kind: PriorityClass

  name: high-priority

globalDefault: false
EOF
kubectl patch deployment <name> -p \
  '{"spec":{"template":{"spec":{"priorityClassName":"high-priority"}}}}'

Default StorageClass
Mental model Default StorageClass = the default printer. PVCs that don't name one will use it automatically. Only one StorageClass can be the default.
What breaks
No StorageClass is marked as default. PVCs that omit storageClassName can't provision automatically.
How it manifests
PVCs without explicit storageClassName stay Pending. kubectl get sc shows no (default) annotation.
Scenario: PVCs failing because no default StorageClass exists. Annotate an existing StorageClass as the default. Verify new PVCs without storageClassName provision successfully.
bash — fix
copy
kubectl get storageclass
kubectl annotate sc <name> \
  storageclass.kubernetes.io/is-default-class=true
kubectl get sc  # verify (default) label appears

Local Storage
Mental model Local storage = data lives on one specific node. Fast I/O, but that pod is permanently tied to that node. Node fails = data unavailable.
What breaks
Local provisioner not running or PV nodeAffinity doesn't match any node. Local PVCs can't bind.
How it manifests
PVC stuck in Pending. Pod scheduled to wrong node. Local path doesn't exist on the target node.
Scenario: Local storage provisioning isn't working. Verify the local-path-provisioner is running, the node path exists, and PV nodeAffinity matches the correct node. Verify PVC binds and pod runs.

Lab 40 of 40
PVC Expansion
Mental model Expanding a PVC is like adding lanes to a highway. The StorageClass must explicitly allow it. Without permission, the request is silently ignored.
What breaks
PVC resize requested but StorageClass has allowVolumeExpansion: false (or missing). Resize never applies.
How it manifests
PVC storage size stays unchanged after edit. Events show "volume expansion not permitted." Disk usage maxes out.
Scenario: PVC needs to grow but resize is blocked. Enable allowVolumeExpansion: true on the StorageClass, then edit the PVC to request more storage. Verify expanded capacity is available to the pod.
bash — fix
copy
kubectl get storageclass
kubectl patch sc <name> \
  -p '{"allowVolumeExpansion":true}'
kubectl edit pvc <pvc-name>
# change spec.resources.requests.storage to new size
kubectl get pvc <pvc-name>  # verify new capacity
