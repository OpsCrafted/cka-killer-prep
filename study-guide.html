Kubernetes Architecture

Kubernetes is a container orchestration platform. The cluster is split into two planes: the control plane (the brain) and worker nodes (the muscle).

Control Plane
kube-apiserver — all cluster communication goes through here
etcd — distributed key-value store, the source of truth
kube-scheduler — assigns pods to nodes
kube-controller-manager — reconciliation loops
cloud-controller-manager — cloud provider integration
Worker Node
kubelet — agent that runs on every node, talks to apiserver
kube-proxy — handles pod networking rules (iptables/ipvs)
container runtime — containerd or CRI-O (not Docker directly)
pods — the actual workloads running here
CKA Exam Note Know the exact names and responsibilities of each component. You will be asked to troubleshoot failing components by name — e.g. "the scheduler is not running, pods are Pending".

Core Objects

Pod
Smallest deployable unit. One or more containers sharing network + storage. Ephemeral — never manually restart a Pod, fix the controller.
Deployment
Manages ReplicaSets. Handles rolling updates and rollbacks. Always use Deployments for stateless apps, not bare Pods.
Service
Stable network endpoint for a set of Pods. Types: ClusterIP (internal), NodePort (external via node), LoadBalancer (cloud LB).
Namespace
Virtual cluster within a cluster. Scope for RBAC, resource quotas. Default namespaces: default, kube-system, kube-public.

Setting Up Your Practice Environment

For CKA prep you need a multi-node cluster. Your aks-project on AKS is perfect. You can also spin up a local kind or k3s cluster for offline practice.

kind — local multi-node cluster

kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4

- role: control-plane
- role: worker
- role: worker

# Create the cluster
kind create cluster --config kind-config.yaml --name cka-lab

# Verify
kubectl get nodes
Exam Tip The CKA exam gives you 6 clusters. You switch between them with kubectl config use-context. Practice this. Forgetting to switch context is one of the most common mistakes.

Bootstrapping a Cluster with kubeadm

kubeadm is the official tool to bootstrap a production-grade cluster. The flow is: init the control plane → configure kubectl → install a CNI → join worker nodes.

kubeadm init flow
# 1. Init control plane (run on control-plane node)
kubeadm init \
  --pod-network-cidr=10.244.0.0/16 \
  --apiserver-advertise-address=<control-plane-ip>

# 2. Set up kubeconfig
mkdir -p $HOME/.kube
cp /etc/kubernetes/admin.conf $HOME/.kube/config

# 3. Install CNI (Flannel example)
kubectl apply -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml

# 4. Join worker nodes (command printed by kubeadm init)
kubeadm join <control-plane-ip>:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash>
CKA Exam Note You may be asked to join a node to an existing cluster. If the token has expired: kubeadm token create --print-join-command

RBAC — Role-Based Access Control

RBAC controls who can do what in the cluster. Four key objects: Role, ClusterRole, RoleBinding, ClusterRoleBinding.

Role / ClusterRole
Defines permissions (verbs on resources). Role = namespace-scoped. ClusterRole = cluster-wide OR can be bound namespace-scoped.
RoleBinding / ClusterRoleBinding
Grants a Role to a user/group/serviceaccount. RoleBinding = namespace. ClusterRoleBinding = cluster-wide.
RBAC — create and bind imperatively
# Create a Role allowing pod reads in 'dev' namespace
kubectl create role pod-reader \
  --verb=get,list,watch \
  --resource=pods \
  --namespace=dev

# Bind it to a user
kubectl create rolebinding read-pods \
  --role=pod-reader \
  --user=jane \
  --namespace=dev

# Verify what jane can do
kubectl auth can-i list pods --as=jane -n dev

Helm & Kustomize

Helm is the package manager for Kubernetes. Kustomize is a template-free overlay system built into kubectl.

Helm essential commands
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm search repo nginx
helm install my-nginx bitnami/nginx --namespace web --create-namespace
helm list -A
helm upgrade my-nginx bitnami/nginx --set replicaCount=3
helm rollback my-nginx 1
helm uninstall my-nginx -n web
Exam Tip For Kustomize: kubectl apply -k ./dir applies a kustomization directory. You don't need to memorise kustomization.yaml syntax deeply — focus on what the exam might ask: applying an overlay, changing a namespace, or setting a label.

Deployments & Rolling Updates

Deployments manage ReplicaSets. The key mechanisms are: rolling update (zero-downtime, default), Recreate (all-at-once), rollback, and scaling.

Deployment operations
kubectl create deployment web --image=nginx:1.21 --replicas=3

# Rolling update
kubectl set image deployment/web nginx=nginx:1.23

# Watch rollout progress
kubectl rollout status deployment/web

# Rollback
kubectl rollout undo deployment/web

# Rollback to specific revision
kubectl rollout undo deployment/web --to-revision=2

# Pause/resume rollout
kubectl rollout pause deployment/web
kubectl rollout resume deployment/web

ConfigMaps & Secrets

ConfigMap + Secret usage
# Create ConfigMap
kubectl create configmap app-config \
  --from-literal=ENV=production \
  --from-literal=LOG_LEVEL=info

# Create Secret (base64 encoded automatically)
kubectl create secret generic db-creds \
  --from-literal=password=supersecret

# Use in pod spec
env:
- name: ENV
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: ENV

Scheduling: Taints, Tolerations & Affinity

Taints & Tolerations
Taints repel pods from nodes. Tolerations allow pods to be scheduled on tainted nodes. Effects: NoSchedule, PreferNoSchedule, NoExecute.
Node Affinity
Attract pods to nodes based on labels. Required (hard rule) vs Preferred (soft rule). More expressive than nodeSelector.
Resource Requests/Limits
requests = guaranteed minimum. limits = hard cap. Scheduler uses requests to fit pods. OOMKiller uses limits to evict.
HPA
HorizontalPodAutoscaler scales replicas based on CPU/memory metrics. Requires metrics-server. Target: average utilization.
CKA Exam Note You will likely be asked to schedule a pod on a specific node using node affinity or nodeSelector, or to taint a node and add a toleration to a pod. Practice writing these specs from scratch.

Pod-to-Pod Connectivity

Every pod gets a unique IP. Pods on different nodes can communicate directly — no NAT. This is guaranteed by the CNI plugin (Flannel, Calico, Cilium, etc.).

Mental model Think of all pods as being on one big flat network, regardless of which node they're on. The CNI is the plumbing underneath that makes this illusion real.

Service Types

ClusterIP
Default. Stable internal IP. Only reachable within the cluster. Use for service-to-service communication.
NodePort
Exposes the service on a static port on every node (30000–32767). Reachable externally via NodeIP:NodePort.
LoadBalancer
Provisions a cloud load balancer. In bare-metal: use MetalLB. Extends NodePort.
ExternalName
Maps a service to a DNS name. No proxying — just a CNAME. Useful for external services.

Network Policies

NetworkPolicies control pod-to-pod traffic. By default, all traffic is allowed. Once you apply a NetworkPolicy to a pod, all traffic not explicitly allowed is denied.

Deny all ingress, allow only from app=frontend
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend
  namespace: backend
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
CKA Exam Note NetworkPolicy YAML cannot be generated imperatively — you must write it from scratch. Memorise the structure: podSelector → policyTypes → ingress/egress → from/to. The exam often asks you to allow traffic on a specific port from a specific namespace.

CoreDNS

CoreDNS is the cluster DNS. Every service gets a DNS name: <service>.<namespace>.svc.cluster.local. Pods: <pod-ip-dashes>.<namespace>.pod.cluster.local.

Debugging DNS kubectl run test --image=busybox --rm -it -- nslookup kubernetes — quick DNS check from inside the cluster.

The Storage Hierarchy

Mental model Think of it like renting a flat. StorageClass = the property developer. PersistentVolume = the actual flat. PersistentVolumeClaim = your rental agreement. Pod = you living in it.
Volume
Ephemeral or durable storage attached to a pod. Lives with the pod. Types: emptyDir, hostPath, configMap, secret, projected.
PersistentVolume (PV)
Cluster-level storage resource. Pre-provisioned (static) or dynamically created by a StorageClass. Has capacity, access modes, reclaim policy.
PersistentVolumeClaim (PVC)
Namespace-scoped request for storage. Bound to a PV that satisfies its requirements. Pod references the PVC, not the PV directly.
StorageClass
Defines a provisioner and parameters. Enables dynamic PV creation when a PVC is submitted. AKS uses azure-disk, azure-file, etc.

Access Modes

ModeAbbreviationDescription
ReadWriteOnceRWORead/write by a single node. Most common.
ReadOnlyManyROXRead-only from many nodes.
ReadWriteManyRWXRead/write from many nodes. Requires NFS/Azure Files/etc.
ReadWriteOncePodRWOPRead/write by a single pod. Kubernetes 1.22+.
PV + PVC + Pod — static provisioning
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-data
spec:
  capacity:
    storage: 1Gi
  accessModes: [ReadWriteOnce]
  persistentVolumeReclaimPolicy: Retain
  hostPath:
    path: /mnt/data
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-data
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 1Gi
CKA Exam Note Know how to: create a PVC, mount it in a pod, check if it is Bound. Also know what happens when a PV is Released (Retain policy = PV is not auto-reused, must be manually cleared).

The Troubleshooting Methodology

Mental model — the funnel Start wide (what's wrong?), then narrow (which layer?), then precise (what exactly failed?). Cluster → Node → Pod → Container → Application.
Step 1: Observe
kubectl get & describe. What is the state? What do events say? Never start fixing before you've read the events.
Step 2: Logs
kubectl logs. If container is dead: --previous. Check app logs, not just Kubernetes events.
Step 3: Exec
kubectl exec into a running pod. Check filesystem, env vars, network connectivity from inside the container.
Step 4: System
For node/component issues: systemctl status kubelet, journalctl -u kubelet, crictl ps/logs.

Troubleshooting Services & Networking

Service not routing traffic — debug steps
# 1. Does the service exist and have the right selector?
kubectl describe svc <name>

# 2. Are there endpoints? (no endpoints = selector mismatch)
kubectl get endpoints <name>

# 3. Are the target pods running and Ready?
kubectl get pods -l <selector-labels>

# 4. Test from inside cluster
kubectl run test --image=busybox --rm -it -- wget -O- <service-name>:<port>

# 5. Check kube-proxy is running
kubectl get pods -n kube-system | grep kube-proxy
CKA Exam Note The #1 service troubleshooting trap: the service selector labels don't match the pod labels. Always check kubectl get endpoints — if it shows <none>, it's a label mismatch. Fix the selector or the pod labels.
Exercise 1
Fix a CrashLoopBackOff pod
Medium
Create a pod using image nginx:broken-tag (intentionally wrong). Observe it fail. Identify the exact error. Fix it by updating the image to nginx:latest. Verify it runs.
kubectl run broken --image=nginx:broken-tag
kubectl get pod broken — ImagePullBackOff
kubectl describe pod broken — Events show failed image pull
Fix: kubectl set image pod/broken broken=nginx:latest
Or delete and recreate with correct image.
Exercise 2
Fix a broken service
Hard
Create a deployment with label app=web. Create a service that selects app=frontend (intentionally wrong). Diagnose why the service has no endpoints. Fix the selector. Test connectivity.
kubectl get endpoints <svc> — shows <none>
kubectl describe svc <svc> — shows wrong selector
kubectl edit svc <svc> — fix selector to app=web
kubectl get endpoints <svc> — now shows pod IPs
Exercise 3
Node is NotReady
Hard
Simulate a NotReady node by stopping the kubelet service on a worker node. Observe the node status in kubectl. SSH to the node, identify the stopped service, restart it, and verify the node returns to Ready.
On worker node: systemctl stop kubelet
On control plane: kubectl get nodes — worker shows NotReady after ~40s
kubectl describe node <worker> — conditions show kubelet stopped
SSH to worker: systemctl start kubelet
systemctl enable kubelet — prevent recurrence
kubectl get nodes — back to Ready
Exercise 4
Decode a failed static pod
Hard
Introduce a deliberate error in the kube-scheduler static pod manifest (e.g. change the image to an invalid tag). Observe what happens to pending pods. Fix the manifest and verify the scheduler recovers.
vi /etc/kubernetes/manifests/kube-scheduler.yaml — break image tag
kubectl get pods -n kube-system — scheduler pod fails/restarts

kubectl get pods -n kube-system — scheduler back to Running
